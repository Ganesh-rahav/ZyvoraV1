## 1. Executive Summary

This document specifies the logical and physical database schema design for the Zyvora MVP. It provides a detailed blueprint of all data entities, relationships, attributes, indexing structures, Row-Level Security (RLS) configurations, and data retention policies for the production environment hosted on Supabase (PostgreSQL).

To align with structural scalability requirements, this database design leverages relational structures to capture user logs (workouts, weights, nutrition), configuration templates, transactional parameters, and the vector-embedded long-term memory of the AI Coach. It serves as the foundation for the upcoming API Specification (`docs/05-api-specification.md`).

---

## 2. Database Design Principles

### Normalization
The core schema is structured up to Third Normal Form (3NF) to minimize redundancy and prevent write anomalies. Redundancy is introduced selectively through computed caches (e.g., aggregate workout summaries) only where query optimization is proven necessary by traffic demands.

### Scalability
The schema leverages PostgreSQL partitioning for high-volume logs (e.g., `analytics_events`, `workout_sessions`) to maintain sub-second index query execution as the dataset grows over time.

### Security
Row-Level Security (RLS) is applied to all tables containing user-owned data. Direct database access is gated; standard application requests run under security contexts containing JWT claims mapped to explicit tenant ownership (`user_id`).

### Performance
All foreign keys are indexed by default. Performance indexes are planned for time-series lookups. The `pgvector` index is optimized via an approximate nearest neighbor algorithm (HNSW) for fast semantic search responses.

### Data Integrity
Referential integrity is guaranteed through database-level foreign key constraints, check constraints (e.g., ensuring numeric values are non-negative), and cascade settings.

### Auditability
All modifications to transactional and subscription states are tracked using historical change logs generated automatically via system triggers.

## 3. Entity Relationship Overview

The Zyvora database is anchored on the `users` table, which maps directly to the Supabase Authentication schema. 

- **User Context Core:** A `user` has a one-to-one relationship with `profiles`, `settings`, and `subscriptions`.
- **Onboarding and Goals:** A `user` has many `goals` and `physique_photos`.
- **AI Analysis:** A `physique_photo` set maps to a single `physique_analysis` generated by vision APIs.
- **Plans & Schedules:** A `user` has many `workout_plans` and `nutrition_plans`.
- **Execution Logging:** A `workout_plan` links to many `workout_sessions`, which aggregate many `exercise_history` logs. A `nutrition_plan` maps to many `meals`, which are populated by `food_items`.
- **Subjective Metrics:** A `user` logs daily physical metrics in `progress_logs` and periodic measurements in `measurements`.
- **AI Intelligence:** The `coach_memory` (semantic vectors) and `chat_history` (message transcripts) belong to a `user`.
- **System and Governance:** `notifications` track alerts, while `audit_logs` track write operations. `analytics_events` exist as an append-only analytical dataset.

## 4. Tables

### 4.1 Users (users)
* **Purpose:** Serves as the public profile representation linked to Supabase's internal auth engine.
* **Schema Definition:**
  | Column Name | Data Type | Constraints / Keys | Description |
  |-------------|-----------|--------------------|-------------|
  | id | UUID | Primary Key, FK (auth.users.id) | Unique user identifier matching authentication. |
  | email | VARCHAR(255) | Unique, Not Null | Verified email address. |
  | created_at | TIMESTAMPTZ | Default NOW(), Not Null | Timestamp when user record was generated. |
  | onboarding_status | VARCHAR(50) | Not Null, Default 'incomplete' | Enum tracking onboarding state. |
* **Indexes:** Primary Key index on `id`.
* **Constraints:** Check constraint limiting `onboarding_status` to ('incomplete', 'completed').
* **Relationships:** One-to-one with `profiles`, `settings`, `subscriptions`. One-to-many with plans, photos, and logs.
* **Soft Delete Strategy:** Complete purge triggered recursively on account deletion.
* **Versioning Strategy:** None.
* **Retention Policy:** Retained for the lifetime of the active account.

### 4.2 Profiles (profiles)
* **Purpose:** Stores biological demographics and baseline physical stats required for program generation.
* **Schema Definition:**
  | Column Name | Data Type | Constraints / Keys | Description |
  |-------------|-----------|--------------------|-------------|
  | id | UUID | Primary Key, FK (users.id) | Matches owner ID. |
  | full_name | VARCHAR(100) | Not Null | Display name. |
  | date_of_birth | DATE | Not Null | Used to calculate age dynamic factors. |
  | biological_sex | VARCHAR(10) | Not Null | Biological gender marker ('male', 'female'). |
  | height | NUMERIC(5,2) | Not Null | Stored in metric centimeters. |
  | current_weight | NUMERIC(5,2) | Not Null | Stored in metric kilograms. |
  | target_weight | NUMERIC(5,2) | Not Null | Stored in metric kilograms. |
  | fitness_level | VARCHAR(20) | Not Null | Level ('beginner', 'intermediate', 'advanced'). |
  | equipment_access | VARCHAR(30) | Not Null | Available tools ('gym_full', 'home_dumbbells', etc.). |
  | injuries_list | TEXT[] | Default '{}' | Active injury descriptions. |
  | dietary_restrictions | TEXT[] | Default '{}' | Active allergies or food limits. |
  | updated_at | TIMESTAMPTZ | Default NOW() | Timestamp of last modification. |
* **Indexes:** Primary Key on `id`.
* **Constraints:** Check constraints verifying `biological_sex` in ('male', 'female'); weights and height > 0.
* **Relationships:** One-to-one with `users`.
* **Soft Delete Strategy:** Cascaded delete with user record.
* **Versioning Strategy:** Changes trigger snapshot logging in `audit_logs`.
* **Retention Policy:** Lifetime of active account.

### 4.3 Goals (goals)
* **Purpose:** Tracks current and historical goals of the user.
* **Schema Definition:**
  | Column Name | Data Type | Constraints / Keys | Description |
  |-------------|-----------|--------------------|-------------|
  | id | UUID | Primary Key, Default uuid_generate_v4() | Unique goal record ID. |
  | user_id | UUID | FK (users.id), Not Null | Identifies target user. |
  | primary_goal | VARCHAR(50) | Not Null | Goal category. |
  | target_date | DATE | Nullable | Expected completion date. |
  | is_active | BOOLEAN | Default TRUE, Not Null | Flags current goal target. |
  | created_at | TIMESTAMPTZ | Default NOW(), Not Null | Creation timestamp. |
* **Indexes:** Indexes on `user_id` and `is_active`.
* **Constraints:** Check constraint verifying `primary_goal` is in ('fat_loss', 'muscle_gain', 'recomposition', 'general_fitness'). Only one active goal allowed per user at a time (enforced via partial unique index).
* **Relationships:** Many-to-one with `users`.
* **Soft Delete Strategy:** Standard delete.
* **Versioning Strategy:** Historical changes recorded by creating a new goal record and setting the previous active goal to `is_active = FALSE`.
* **Retention Policy:** Historical goals preserved indefinitely to support context learning.

### 4.6 WorkoutPlans (workout_plans)
* **Purpose:** Holds structured, AI-generated training plans defining multi-week workout parameters.
* **Schema Definition:**
  | Column Name | Data Type | Constraints / Keys | Description |
  |-------------|-----------|--------------------|-------------|
  | id | UUID | Primary Key | Unique plan record ID. |
  | user_id | UUID | FK (users.id), Not Null | Associated user. |
  | block_number | INT | Not Null | Sequenced block tag (e.g., 1). |
  | start_date | DATE | Not Null | Start of programming execution. |
  | end_date | DATE | Not Null | Projected program completion. |
  | frequency_days | INT | Not Null | Scheduled sessions per week. |
  | is_active | BOOLEAN | Default TRUE, Not Null | Flags current program block. |
  | rationale | TEXT | Not Null | AI's explanation of plan design. |
* **Indexes:** Composite index on `(user_id, is_active)`.
* **Constraints:** Check constraint ensuring `end_date` is later than or equal to `start_date`.
* **Relationships:** Many-to-one with `users`. One-to-many with `workout_sessions`.
* **Soft Delete Strategy:** Cascaded delete.
* **Versioning Strategy:** Active status switches to false when new plans are generated.
* **Retention Policy:** Retained indefinitely to record progressive volume over time.

### 4.15 CoachMemory (coach_memory)

* **Retention Policy:** Maintained indefinitely to build the user profile memory.

## 6. Indexing Strategy

To maintain target database load times (<1.5 seconds) at production scale, the following index architecture is configured:

### Performance Indexes
* **Foreign Keys:** B-Tree indexes on all foreign key columns (e.g., `user_id`, `plan_id`, `session_id`) to optimize join queries.
* **Time-Series Ranges:** Composite indexes incorporating timestamp columns (e.g., `(user_id, logged_date)` on `progress_logs`) to optimize history chart loading.

### Search Indexes
* **Exercise Lookup:** B-Tree index on `primary_muscle_group` in `exercises` for fast filter operations in search panels.
* **Vector Search:** HNSW index with Cosine Distance on the `vector_embedding` column in `coach_memory` to enable sub-second semantic retrieval.

### Unique Indexes
* **Daily Metric Uniqueness:** Composite unique index on `(user_id, logged_date)` in `progress_logs` to enforce one metric entry per day.
* **Measurement Uniqueness:** Composite unique index on `(user_id, measured_date)` in `measurements` to ensure one measurement record per day.

## 7. Row Level Security

Supabase Row-Level Security (RLS) is applied to enforce data isolation:

### Base Policy Rules
* All tables hosting user data (e.g., `profiles`, `workout_plans`, `progress_logs`, `chat_history`) must have RLS activated.
* Standard user access policies are defined by the expression: `auth.uid() = user_id`.
* The `users` and `settings` tables configure self-access checks: `auth.uid() = id`.
* Unauthenticated database access is blocked by default across all user tables.

### Admin Overrides
* Service-role connections bypass RLS policies to allow administrative actions.
* The `admin_users` table restricts query access to verified administrators (`auth.uid() = id`).
* Read-only public tables (e.g., `exercises`) define access policies permitting public reads but limiting write access to administrative keys.

## 9. Audit Logging

### Activity Triggers
* Trigger functions execute after write actions on `users`, `profiles`, and `subscriptions`.
* The database writes the operation name ('INSERT', 'UPDATE', 'DELETE') and column variations (JSON comparison of `old_data` vs. `new_data`) to the `audit_logs` table.

### Access Restrictions
* Standard user accounts have no database read permissions on `audit_logs`.
* Audit log records are restricted to verified administrators (`admin_users`) for security review and support diagnostics.

## 10. Migration Strategy

### Schema Migrations
* Database updates are tracked as incremental, numbered SQL scripts.
* Schema evolution rules dictate that column modifications are applied as non-breaking mutations (e.g., adding nullable columns first, then running backfills before applying Not Null constraints).

### Verification
* Migration scripts are tested on a local database instance before being promoted to staging.
* Rolling back updates utilizes corresponding rollback scripts to restore the schema version state without data loss.

## 12. Risks

* **Vector Search Latency:** Growing vector datasets could degrade search speeds in the `coach_memory` table.
  * *Mitigation:* Configure index parameter tuning (`ef_search` and `m`) and run semantic searches within partitioned `user_id` subsets.
* **RLS Overhead:** Complex RLS policies on frequently queried tables could degrade page load speeds.
  * *Mitigation:* Use helper functions within policy checks and keep user checks simple.
* **Partition Maintenance:** Failures in partition generation scripts could block analytical logs.
  * *Mitigation:* Implement pre-generation jobs that create tables multiple months in advance.
